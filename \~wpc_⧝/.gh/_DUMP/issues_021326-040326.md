Let me restate your architecture vision to confirm I'm tracking it cleanly:

The Core Idea:

A git-native, plain-text mirror of all GitHub collaborative metadata (issues, comments, milestones, labels, projects, PRs) stored in a dedicated branch (or parallel remote), such that:

The mirror is continuously synced from GH during normal operation
In event of takedown, the mirror becomes the source of truth
Local text-file manipulation of that mirror maintains feature parity with GH native tooling
On restoration, the mirror can be used to reconstitute GH state via API
The whole thing is portable across git remotes — no GH dependency to operate it
What this actually requires, broken into layers:

Layer 1 — Schema/Format (the hardest design decision) The plain-text representation needs to be both human-editable and round-trippable back to GH API format. Candidates:

YAML front-matter + Markdown body per issue/comment (most human-friendly)
JSON (lossless, less human-editable)
TOML (middle ground)
The key insight: you want a format where git diff on the mirror branch is meaningful — i.e., a new comment reads like a new comment, not a JSON blob mutation.

Layer 2 — Sync (GH → local) python-github-backup or gh CLI scripted dumps are adequate here. The important thing is the sync writes to an orphan branch (no working tree pollution) and commits are timestamped + attributed.

Layer 3 — Offline operation This is where it gets interesting. You want tooling that can read and write the mirror format as if it were a lightweight issue tracker. This is essentially what git-bug does natively — but the question is whether git-bug's sync fidelity with GH is sufficient for your reconstitution needs.

Layer 4 — Reconstitution (local → GH via API) This is the rarest but most critical layer. You need a script/tool that can walk the mirror and replay state to a fresh GH repo via API — respecting ordering, authorship (where possible), and linkages (issue references, PR associations).

Honest tool assessment against your requirements:

Tool | Sync | Offline Op | Reconstitute | Round-trip fidelity -- | -- | -- | -- | -- python-github-backup | ✅ | ❌ (read-only JSON) | 🟡 (scriptable) | High git-bug | 🟡 (GH sync partial) | ✅ | 🟡 | Medium gh CLI + custom scripts | ✅ | ❌ native | ✅ (scriptable) | Highest Custom YAML/MD + gh CLI | ✅ | ✅ (with editor) | ✅ | You define it
My recommendation for your use case:

A thin custom layer built on gh CLI + structured Markdown/YAML in an orphan branch, with two scripts:

mirror-pull — dumps GH state → structured files, commits to mirror branch
mirror-push — reads mirror branch → replays to GH API on target repo
This keeps the toolchain minimal, the format human-readable and git diff-friendly, and the reconstitution path explicit and auditable.
